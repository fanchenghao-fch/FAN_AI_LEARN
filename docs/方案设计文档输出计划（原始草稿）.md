## 1. 技术选型

### 1.1 小程序前端框架 todo 目前不用太关注前端页面和 UI 设计，后续我会单独设计开发网页原型，然后再做小程序

#### 1.1.1 候选方案概览

微信小程序开发目前有三条主流路线：

| 路线 | 技术栈 | 代表框架 | 跨端能力 |
|------|--------|---------|---------|
| **原生开发** | WXML / WXSS / JS | — | 仅微信 |
| **编译时方案** | Vue SFC → 编译为 WXML | uni-app (DCloud) | 微信+支付宝+抖音+H5+App |
| **运行时方案** | React/JSX + 虚拟DOM → setData | Taro 3.x (京东) | 微信+支付宝+抖音+H5+RN |

#### 1.1.2 深度对比（基于 2026 年最新数据）

| 维度 | 原生开发 | uni-app | Taro 3.x |
|------|:---:|:---:|:---:|
| **最新版本** | 基础库持续迭代 | uni-app x (UTS) + Vue 3 | Taro 3.6.x（Taro 4 开发中） |
| **技术栈** | WXML/WXSS/JS | Vue 2/3 | **React** / Vue / Preact |
| **TypeScript** | ⭐⭐⭐ 基础支持 | ⭐⭐⭐⭐ 可用 | ⭐⭐⭐⭐⭐ **原生支持极佳** |
| **IDE** | 微信开发者工具 | HBuilderX / VSCode | **VSCode / WebStorm** |
| **首屏渲染 (500条列表)** | ⭐⭐⭐⭐⭐ 320ms | ⭐⭐⭐⭐⭐ 380ms | ⭐⭐⭐⭐ 450ms |
| **包体积增量** | 基准 | +80~120KB | +150~200KB |
| **React 生态复用** | ❌ | ❌ | ✅ Zustand/Jotai/React Query 等 |
| **组件库生态** | WeUI | uView / uni-ui | **NutUI / TDesign** |
| **条件编译** | 无（仅微信） | ⭐⭐⭐⭐⭐ 全文件覆盖 | 通过 `process.env.TARO_ENV` |
| **学习曲线** | 低（但技术孤立） | 低（Vue + 傻瓜式） | 中（需前端工程化能力） |
| **2026 市场占比** | ~15% | ~45% | ~35% |
| **社区活跃度** | 微信官方 | 极活跃（插件市场数万） | 活跃（京东+社区驱动） |

#### 1.1.3 架构原理差异

```
uni-app（编译时方案）：
  .vue 文件 → SFC解析 → AST中间抽象层 → 各平台模板生成器（WXML/AXML/DOM）
  → 小程序端不运行 Vue，编译期静态展开 → 输出代码接近原生

Taro 3（运行时方案）：
  React/JSX → 虚拟DOM → Taro Runtime(模拟DOM/BOM) → setData → 小程序渲染
  → 多一层虚拟 DOM → setData 转换，代价是有可感知的性能损耗
  → Taro 4 将引入「混合编译模式」缩小差距
```

#### 1.1.4 框架推荐：**Taro 3.x + React**

**推荐理由**：

1. **React 生态复用价值巨大**
   - 本项目有大量状态管理需求（答题状态、连击计数、进度追踪、错题管理），React Hooks（`useReducer`、`useContext`）天然适合
   - Zustand 等轻量状态库可在 Taro 中无缝使用

2. **TypeScript 支持业界最佳**
   - 本项目的核心数据结构（题目、答题记录、分析报告）类型复杂，需要强类型约束
   - Taro 的类型定义完善，IDE 智能提示极佳

3. **标准 npm 工程化**
   - VSCode 开发体验一致，无需绑定 HBuilderX
   - 支持标准 ESLint/Prettier/Git Hooks 工作流

4. **性能差距可控**
   - 本项目非长列表/直播间等高性能场景，常规答题交互的性能差异用户无感
   - 450ms vs 320ms 的首屏差异对答题类应用影响极小

5. **迁移灵活性**
   - 未来如果需要 H5 版本（如管理后台），Taro 可复用大量逻辑代码

**不选 uni-app 的原因**：
- 本项目非 Vue 技术栈，uni-app 的编译时方案在复杂交互（实时答题状态流转）中存在动态特性受限问题
- HBuilderX 强绑定不符合标准前端工作流

**不选原生开发的原因**：
- 无法复用 React 生态，状态管理、网络请求等需从零搭建
- 技术孤立，团队成长性差
- 未来如需跨端（H5 管理后台），需完全重写

### 1.2 后端 / BaaS 方案

#### 1.2.1 候选方案对比

| 维度 | 微信云开发 | 自建后端 (FastAPI) | 自建后端 (Node.js/NestJS) |
|------|-----------|-------------------|--------------------------|
| **运维成本** | 零运维 | 需服务器 + 域名 + SSL | 同左 |
| **AI 集成** | 内置 `wx.cloud.extend.AI`，原生调用 DeepSeek/混元/GLM/Kimi | 需自行封装各模型 API | 同左 |
| **数据库** | 云数据库（MongoDB 兼容） | PostgreSQL / MySQL | 同左 |
| **文件存储** | 内置云存储 | 需对接 OSS/COS | 同左 |
| **冷启动** | 2026 年优化至毫秒级 | 无（常驻） | 无（常驻） |
| **成本** | 按量付费，免费额度覆盖 MVP | 服务器月费 ¥100-500+ | 同左 |
| **扩展性** | 中等（长期可能受限） | 高 | 高 |
| **微信生态集成** | ⭐⭐⭐⭐⭐ 原生集成 | ⭐⭐⭐ 需自行对接 | ⭐⭐⭐ |
| **AI Token 补贴** | 2026 年微信送 1 亿混元 Token | 无 | 无 |

#### 1.2.2 推荐方案：**MVP 阶段使用微信云开发，增长期迁移自建后端**

**MVP 阶段（Phase 0-2）：微信云开发**
- 零运维成本，1 人可完成全栈
- `wx.cloud.extend.AI` 直接调用多模型，无需自建 API 网关
- 微信「AI 应用成长计划」提供免费 Token 和流量扶持
- 云函数处理出题逻辑，云数据库存储题目和记录

**增长期（Phase 3+）：混合架构 / 迁移自建后端**
- 当 DAU 破万、AI API 调用量增大时，自建后端降低成本
- 推荐 **Python FastAPI**（AI 生态最成熟）+ PostgreSQL + Redis
- 云开发保留作为微信生态入口（登录、支付、订阅消息）

### 1.4 状态管理

**推荐：Zustand**

| 候选 | 包大小 | 学习曲线 | Taro 兼容性 | 适用场景 |
|------|:---:|:---:|:---:|------|
| **Zustand** | ~1KB | 极低 | ✅ 完美 | **推荐** — 轻量、无 boilerplate |
| Jotai | ~2KB | 低 | ✅ | 原子化状态，适合细粒度更新 |
| Redux Toolkit | ~11KB | 高 | ✅ | 过度设计，不适合小程序 |

**选 Zustand 的理由**：
- 包体积极小（~1KB），不增加小程序包体积压力
- API 极简，无 Provider 包裹，无 action/reducer 模板代码
- 天然支持 Taro（纯 JS 库，无 DOM 依赖）
- 支持 `persist` 中间件，可持久化答题状态到 Storage

### 1.6 海报生成

**推荐：Painter**

| 候选 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **Painter** | 复杂动态海报 | JSON 配置、自排版、自动换行 | 额外依赖 |
| wxml-to-canvas | 简单固定海报 | 微信官方、WXML 语法 | 排版能力弱 |
| wxml2canvas-2d | 所见即所得 | 直接复用 WXML | 不支持 Skyline |

**选 Painter 的理由**：
- 本项目海报元素动态变化（分数、正确率、知识领域、金句），Painter 的 JSON 配置 + 自动排版最适合
- 支持圆角、阴影、自动换行、动态高度
- 自动处理网络图片下载和缓存
- 社区活跃，有可视化编辑器辅助开发

### 1.7 流式响应方案

小程序不支持浏览器标准的 `EventSource` / `fetch streaming`，需要特殊处理：

#### 方案 A（推荐 MVP）：微信云开发 `extend.AI.streamText()`

```js
const ai = wx.cloud.extend.AI
const provider = ai.createModel("cloudbase")

const res = await provider.streamText({
  data: {
    model: "deepseek-v4-pro",
    messages: [{ role: "user", content: "生成10道关于中国近代史的选择题" }]
  }
})

// 逐字流式接收
for await (let str of res.textStream) {
  // 追加到页面显示
}
```

- ✅ 零配置，微信原生封装
- ✅ 自动处理 SSE 解析、重连、错误
- ❌ 依赖云开发环境

#### 方案 B（增长期备选）：`wx.request` + `enableChunked`

当自建后端时使用。关键配置：

```js
wx.request({
  url: 'https://api.your-domain.com/quiz/generate',
  method: 'POST',
  enableChunked: true,         // 开启分块传输
  responseType: 'arraybuffer', // 二进制模式防中文乱码
  enableHttp2: false,          // 关键：避坑 HTTP/2 协议错误
  // ...
})
requestTask.onChunkReceived((res) => {
  // 手动解析 SSE 数据块
})
```

**Nginx 必备配置**（自建后端时）：
```nginx
proxy_buffering off;
proxy_cache off;
proxy_http_version 1.1;
gzip off;
proxy_read_timeout 24h;
```

### 5.2 答题引擎实现要点

```typescript
// 核心答题状态机（Zustand Store 简化示例）

import { create } from 'zustand'

interface QuizStore {
  // 状态
  questions: Question[]
  currentIndex: number
  answers: UserAnswer[]
  combo: number
  maxCombo: number
  status: QuizStatus

  // 操作
  selectAnswer: (answer: string) => void
  skipQuestion: () => void
  goBack: () => void
  nextQuestion: () => void
  completeQuiz: () => void
}

export const useQuizStore = create<QuizStore>((set, get) => ({
  questions: [],
  currentIndex: 0,
  answers: [],
  combo: 0,
  maxCombo: 0,
  status: 'idle',

  selectAnswer: (answer: string) => {
    const { questions, currentIndex, combo, maxCombo } = get()
    const question = questions[currentIndex]
    const isCorrect = answer === question.correctAnswer

    set({
      answers: [...get().answers, {
        questionId: question.id,
        userAnswer: answer,
        isCorrect,
        timeSpent: Date.now() - questionStartTime
      }],
      combo: isCorrect ? combo + 1 : 0,
      maxCombo: isCorrect ? Math.max(maxCombo, combo + 1) : maxCombo
    })
  },

  skipQuestion: () => {
    // 跳过：不记录答案，移到下一题
    set({ currentIndex: get().currentIndex + 1 })
  },

  goBack: () => {
    if (get().currentIndex > 0) {
      set({ currentIndex: get().currentIndex - 1 })
    }
  },

  nextQuestion: () => {
    const { currentIndex, questions } = get()
    if (currentIndex < questions.length - 1) {
      set({ currentIndex: currentIndex + 1 })
    } else {
      get().completeQuiz()
    }
  },

  completeQuiz: () => {
    set({ status: 'completed' })
    // 触发分析报告生成
  }
}))
```


```
├─────────────────────────────────────────────┤

│                                              │
```

## 7. 开发规范与约定

### 7.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | kebab-case | `question-card.tsx` |
| 组件名 | PascalCase | `QuestionCard` |
| 函数/变量 | camelCase | `selectAnswer`, `currentIndex` |
| 常量 | UPPER_SNAKE_CASE | `MAX_QUESTIONS`, `DEFAULT_DIFFICULTY` |
| 类型/接口 | PascalCase | `QuizSession`, `UserAnswer` |
| CSS 类名 | kebab-case 或 BEM | `.quiz-page__option--selected` |

### 7.2 代码组织

- 每个页面独立目录：`.tsx` + `.scss` + `.config.ts`
- 页面专属组件放在 `pages/[name]/components/`
- 页面专属 Hooks 放在 `pages/[name]/hooks/`
- 跨页面共享组件放在 `src/components/`
- AI Prompt 模板独立为 `.js` 文件，统一管理

### 7.3 Git 工作流

```
main         ← 生产分支（受保护）
  ├── develop ← 开发分支
  │   ├── feature/phase0-input     ← 功能分支
  │   ├── feature/phase0-quiz
  │   ├── feature/phase0-result
  │   └── ...
  └── release/v1.0  ← 发布分支
```

### 7.4 Commit 规范

```
feat: 新功能        feat: 添加答题连击特效
fix: 修复bug        fix: 修复选项点击后状态不更新
refactor: 重构      refactor: 抽取答题引擎为独立 Hook
style: 样式调整     style: 调整选项按钮间距
docs: 文档更新      docs: 更新 AI 出题 Prompt
chore: 构建/工具    chore: 升级 Taro 至 3.6.x
```
